## 两个核心问题

长时间运行的代理编码任务存在两个问题：

1. **上下文窗口退化**：当 context 快填满时，模型会表现出”上下文焦虑”（context anxiety）——接近限制时过早结束工作。Compact（上下文压缩）虽然保留了对话连续性，但焦虑感仍然存在。有效的方案是上下文重置（context reset）——彻底清空窗口重新开始，而非仅做压缩。这个方案成为 Harness 工程设计的决策之一。
2. **自我评估偏差**：模型评估自己完成的工作时，打分总是偏向正面——即使在人类看来质量明显一般。这在设计等主观领域尤其突出，但也影响可验证的任务。

## GAN 启发：生成与评估的分离

Anthropic 借鉴自 GAN（生成对抗网络）：将生成和评估分离为独立角色。这样做的原因是：**调优一个独立评估器的怀疑倾向，比让生成器学会自我批判要可行得多**。分离后，评估器可以被单独调校为适度怀疑，而生成器则专注于从具体的外部反馈中迭代改进。

## 前端设计 Harness

在全栈编码之前，作者先在前端验证了这个模式。Claude 的基线倾向是生成”安全、可预测的布局，技术上可用但视觉上平庸”，目标是将其提升为有辨识度的高质量设计。

评估采用四项标准：
- **设计质量**（Design Quality）：色彩、排版、布局和图像的整体协调性
- **原创性**（Originality）：是否有自主设计决策，而非模板默认值；会惩罚典型的”AI 味”输出（如紫色渐变 + 白色卡片）
- **工艺**（Craft）：排版层次、间距、色彩和谐度、对比度等技术执行
- **功能性**（Functionality）：独立于美学的可用性

权重上刻意侧重设计质量和原创性（Claude 的短板），弱化工艺和功能性（Claude 本身就做得不错）。

一个发现：**评分标准的措辞本身就在引导生成方向**。仅仅是在 prompt 中写出这些标准（如”最好的设计是博物馆级别的”），即使不经过评估循环，首轮输出也已经明显优于没有标准的基线。标准不只是度量工具，它同时也是方向盘。

典型案例：设计荷兰博物馆网站的 demo。到第 9 轮迭代，已经是一个精致的深色主题landing页。但在第 10 轮，Generator 做了一次完全的重新构想——转为 3D 空间体验，使用 CSS perspective 构建棋盘格地板、自由摆放的艺术品和门廊式导航。这种创造性跳跃在单次生成中几乎不会出现。

## 三代理全栈编码系统

概览图：
![](https://iili.io/qPoRUAb.png)

### Planner

将之前需要人工编写的规格说明自动化。接收 1-4 句的用户描述，扩展为完整的产品规格。Planner 刻意聚焦于产品愿景和高层技术设计，**回避细粒度的技术实现细节**——因为规划层面的过度指定会在下游实现中级联放大为错误。

例：用户输入”2D 复古游戏制作器”，Planner 将其扩展为包含 16 个特性、横跨 10 个 sprint 的完整规格，涵盖精灵动画、AI 辅助生成、音效和游戏导出等模块。

### Generator

早期设计采用 sprint 模式，每个 sprint 实现一个 feature（one-feature-at-a-time）。技术栈固定为 React + Vite + FastAPI + SQLite（后来改为 PostgreSQL），使用 git 做版本管理。每个 sprint 结束后，Generator 先做一次”自我评估”，再将工作交给 Evaluator 进行 QA。

随着 Opus 4.6 的发布，模型能够在 2 小时以上的构建中保持连贯，sprint 分解不再必要。Generator 转为一次性从 spec 中获取并实现多个 feature，不再需要人为的 sprint 边界。

### Evaluator

Evaluator 使用 Playwright MCP 以真实用户的方式与运行中的应用交互——点击 UI、测试功能、检查 API 端点和数据库状态，而非仅做静态的截图分析。

评分标准覆盖四个维度：**产品深度、功能完整性、视觉设计、代码质量**。每个维度设有硬性及格线，低于及格线则本次实现判定失败，Generator 会收到具体的错误原因反馈。

**但开箱即用的 Claude 其实是个糟糕的 QA agent。** 早期运行中，评估器会识别出合理的问题，然后又自己说服自己忽略掉；测试也流于表面，遗漏边界情况。解决方案是基于日志逐轮分析评估器的判断偏差点，迭代优化 prompt，并用 few-shot 示例校准评分标准。调优后的评估器能捕获诸如”矩形填充工具只在拖拽起止点放置方块而非填满区域”、”删除键处理器缺少必要的条件检查”、”FastAPI 路由顺序导致帧重排端点失效”等具体 bug。

随着模型能力的升级（Opus 4.6），Generator 每次实现的质量更加完善，Evaluator 从每个 sprint 逐次评分，调整为在整个工程尾部做一次整体评估。评估器的价值变得与任务相关——只有当任务超出模型裸能力边界时，QA 环节才真正产生增量价值。

### Sprint 契约机制

每个 sprint 实现之前，Generator 和 Evaluator 会协商出一份”sprint 契约”——在编码开始前就对”什么算完成”达成一致。这弥合了高层用户故事与可测试实现之间的鸿沟。Generator 提出它将构建什么以及如何验证成功，Evaluator 审查该提案以确保方向正确，两者迭代直至达成共识。

### 代理间通信：基于文件的协议

代理之间通过文件系统进行通信：一个代理写入文件，另一个读取后在原文件中回应或写入新文件，前一个代理再读取这个新文件。Generator 完成契约约定的工作后，将结果交给 Evaluator 进行 QA。这种文件交接方式相比基于对话的协调更简单，同时确保工作忠实于规格、避免过早地过度指定实现细节。

20 倍的成本与质量上的差异。

## Harness 与模型的共同进化

Harness 的每个组件都编码了对模型能力边界的假设。当新模型发布时，这些假设需要被重新检验。Anthropic 的原则是：”**找到最简单的可行方案，只在必要时增加复杂度。**”

Opus 4.6 发布后的具体变化：
- **Sprint 分解被移除**：模型能够连贯地处理 2 小时以上的构建，不再需要人为切分
- **Evaluator 后移**：从每个 sprint 评分变为工程尾部整体评分
- **但 QA 依然有价值**：即使用 Opus 4.6，Generator 仍然会遗漏细节或对功能做 stub 处理，评估器仍能捕获有意义的缺陷

结论：Harness 的有趣组合空间并不会随模型进步而缩小——**harness 的边界也在移动**。
AI 工程师的工作是持续找到下一个新的有效组合，而非守住旧的架构。当新模型发布时，需要去除不再对性能起支撑作用的组件，同时添加新组件以触及之前无法达到的能力上限。

## 一些思考

### Harness 的保质期？

Anthropic在 Sonnet 4.5 时代设计的 sprint 分解机制，到 Opus 4.6 发布后就被直接移除了。不是重构，而是直接删除了，因为当初那段代码解决的问题已经不存在了。

保质期这里其实反映了 harness 工程和传统软件工程的一个根本性差异：**harness 的每一个组件本质上都是对模型当前能力缺陷的一次编码**。模型升级，缺陷消失，组件就变成了冗余。传统软件的代码在解决确定性问题，生命周期以年计；而 harness 代码在补偿一个快速移动的目标的不足，生命周期可能只有几个月。

我最初的看法其实是偏"模型派"的，当模型足够强大时，围绕模型的工程会逐渐消失。这个直觉来自我自己的使用体验：从 Sonnet 3.7 一路用过来，早期需要在 prompt 中写大量规则来约束模型行为、避免它犯特定类型的错误，但随着模型升级，这些需要我添加的规在不断减少。模型越来越聪明，犯错越来越少，需要我操心的事情也越来越少。

但最近读了大量的 harness 工程相关的文章后，我认为Harness 工程不会消失，但它需要解决的问题也在迁移——从低级别的能力补偿上移到更高级别的系统性协调（多代理合作、评估标准设计、验收契约协商）。我给模型写的那些逐渐不在需要的规则，和Anthropic逐渐删除的那些 harness 组件，本质上是同一个现象在不同尺度上的体现。

 **harness 必须足够薄**。不是够不够用的问题，更想是一个敏捷性问题。模型的进化速度极快，一个笨重的、深度耦合于当前模型能力的 harness，在下一代模型发布时就变成了负担——你要花大量时间去拆解和重写那些不再需要的逻辑，而不是快速适配新的能力边界。薄的 harness 才能跟上模型的进化节奏。这也是为什么 Claude 工程师说的"**构建六个月不会淘汰的工程**"，不是预测六个月后模型会变成什么样，而是让你的工程足够轻，轻到即使预测错了也不心疼重来。

###  harness 的成本曲线？

Anthropic给出的数字很直观：实现这个游戏编辑器需要单代理 20 分钟 $9，完整 harness 6 小时 $200。20 倍的成本换来质量的代际差异。

但这个倍数本身在变化。Anthropic提到的另一个实现 DAW 案例中，Opus 4.6 + 简化后的 harness 成本降到了 $124.70。两股力量在同时作用：单代理的裸能力在上升（低成本方案的质量天花板在抬高），harness 在变薄（高成本方案的成本在下降）。两条线在收敛。

我当然希望某个时刻单代理就足够好了，实现最简洁的工程形态。但现实地看，我认为这两条线会保持一个差距。原因也很简单：人的期望不是静态的。当模型能独立完成一个"能用"的游戏编辑器时，我们的要求会变成"好用"的；当它能做到"好用"时，我们又会要求"新颖"的。需求的标杆随能力上升而同步上移，harness 在模型裸能力之上叠加的那一层增量价值，可能会一直存在，只是它增量的内容在不断变化。
